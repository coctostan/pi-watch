<objective>
## Goal
Prove — or characterize the exact conditions under which — a `pi`-extension-registered custom tool enters the **active tool set** and is **executable end-to-end** (LLM calls it, receives its result). Produce a reproducible activation recipe that Phase 5 (`watch` tool primitive) will follow.

This de-risks the one load-bearing assumption that is NOT yet spike-proven: during spike #1, a tool registered by a `pi -e ext.ts` extension was advertised but "Tool not found" in print mode (`-p`); the official `dynamic-tools.ts` example reportedly had the same issue.

## Purpose
Everything in pi-watch hinges on the `watch` tool actually running. If a custom tool cannot be activated/executed reliably, the entire tool-first architecture (DESIGN.md §7) is blocked. This is why the roadmap is risk-first: prove the tool can run before building a sampler, router, or adapters for it. Connects to PROJECT.md core value ("the `watch` tool is the primitive") and the PRD Risk "[Load-bearing, un-de-risked] Tool activation".

## Output
- `spikes/01-tool-activation/watch-probe.ts` — minimal throwaway probe extension with a self-diagnostic.
- `spikes/01-tool-activation/run-activation-tests.sh` — model-free test runner across load methods × run modes.
- `spikes/01-tool-activation/activation-report.*.json` — captured diagnostics (generated artifacts).
- `spikes/01-tool-activation/FINDINGS.md` — the authoritative activation recipe + print-mode verdict.
</objective>

## AC-1: Tool registration is visible
```gherkin
Given the watch-probe extension is loaded by pi (any supported method)
When the session starts and the self-diagnostic runs
Then `watch_probe` appears in `pi.getAllTools()`
And the diagnostic records this membership plus `ctx.mode` to an activation-report JSON file
```

## AC-2: Active-set membership is characterized across load methods and modes
```gherkin
Given the probe is loaded via (a) `-e ./spikes/.../watch-probe.ts` and (b) project-local auto-discovery (`.pi/extensions/`)
And pi is run in print mode (`-p`) and (where available) json mode
When the self-diagnostic checks `pi.getActiveTools()` before and after calling `pi.setActiveTools([... , "watch_probe"])`
Then for every combination the report states whether `watch_probe` was in the active set by default, and whether `setActiveTools` makes it active
And any combination reproducing the spike-#1 gotcha (in getAllTools but NOT executable/active) is explicitly identified
```

## AC-3: End-to-end executability is confirmed and the print-mode verdict is recorded
```gherkin
Given a human runs pi with the probe loaded
When the LLM is asked to call `watch_probe` with a known message
Then the tool executes and returns its sentinel result (proving executability), in at least one supported run mode
And FINDINGS.md records, per run mode, one of: WORKS / WORKS-WITH-setActiveTools / KNOWN-LIMITATION(+workaround)
```

## AC-4: A reusable activation recipe exists for Phase 5
```gherkin
Given the spike is complete
When a future plan needs to ship the real `watch` tool
Then FINDINGS.md provides a concrete, copy-pasteable recipe: where the extension must live, how it is loaded, whether `setActiveTools` is required, and which run modes are supported vs limited
```

<task type="auto">
  <name>Task 1: Build the watch-probe extension with a model-free self-diagnostic</name>
  <files>spikes/01-tool-activation/watch-probe.ts</files>
  <action>
    Create a minimal, throwaway probe extension mirroring the structure of dynamic-tools.ts (read that example first).

    Default export `function (pi: ExtensionAPI)` that:
    1. Registers a tool via `pi.registerTool({...})`:
       - name: "watch_probe"
       - label: "Watch Probe"
       - description: "Activation probe for pi-watch. Echoes its message back with a sentinel marker."
       - promptSnippet + promptGuidelines: include them so the tool is advertised (e.g. guideline "Use watch_probe when asked to probe tool activation.").
       - parameters: Type.Object({ message: Type.String({ description: "Message to echo" }) })
       - execute: return content [{ type: "text", text: `WATCH_PROBE_OK:${params.message}` }], details { tool: "watch_probe" }.
    2. On `session_start`, runs a self-diagnostic and writes a JSON report:
       - Read an output path from env var `WATCH_PROBE_REPORT` (fallback `spikes/01-tool-activation/activation-report.json`).
       - Capture: `ctx.mode`, `ctx.hasUI`.
       - `before` = { inAll: getAllTools().some(t=>t.name==="watch_probe"), inActive: getActiveTools().some(...) }.
       - Then call `pi.setActiveTools([...currentActiveNames, "watch_probe"])` (union; do not drop existing tools), and capture `after` = same two booleans.
       - Write JSON: { mode, hasUI, timestamp, before, after, allToolNames, activeToolNames } using node:fs (import { writeFileSync, mkdirSync } from "node:fs"; ensure parent dir exists).
       - Wrap in try/catch; on error, write { error: String(e) } so a failed run still produces a file. Use ctx.ui.notify only when ctx.hasUI.
    3. Register a command `/watch-probe-report` that prints the latest report path/content via ctx.ui.notify (TUI convenience; guard on ctx.hasUI).

    Imports: `import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";` and `import { Type } from "typebox";`. Export the input type: `export type WatchProbeInput = Static<typeof ...>` is optional.

    Avoid: building any real watch/sampler logic; adding npm deps (use only the two documented imports + node:fs); writing into `.paul/`.
  </action>
  <verify>pi -e ./spikes/01-tool-activation/watch-probe.ts -p "say hi" ; test -f spikes/01-tool-activation/activation-report.json && grep -q '"inAll":true' spikes/01-tool-activation/activation-report.json</verify>
  <done>AC-1 satisfied: watch_probe is present in getAllTools() and the report file is written with mode + membership.</done>
