<objective>
## Goal
Replace the escalating `tier2Runner` stub with a real **tier-2 adapter**: an
OpenAI-compatible `/v1/chat/completions` call that sends the sampled frames as
ordered `image_url` blocks + text (DESIGN §4/§5 #2 — frames-as-images, *not* a
video file) and returns the model's answer. The adapter is **baseURL + model id
only** — one code path that serves local Qwen3-VL (`mlx_vlm.server`) or a hosted
OpenAI-compatible endpoint (e.g. Gemini) with no per-model forks. It returns
`null` on missing config / non-2xx / network error / empty response so
`walkTierChain` falls through to tier 3.

## Purpose
Tier 2 is the middle rung of the core value prop — "cheapest path that works":
a native video-capable model that is far cheaper than dumping every frame into
the orchestrator (tier 3) when one is available, while staying local-first and
model-agnostic. The async tier seam shipped in 06-01 exists specifically to
host this network adapter; 06-02 fills it in. After this, tiers 1, 2, and 3 are
all real and the v0.1 tier story is complete.

## Output
- A tier-2 adapter (pure request-builder + response-parser + an effectful runner
  factory + an env-bridge config resolver) wired into `defaultRunners`.
- New/updated exports on the `watch` barrel.
- Deterministic specs using an **injected HTTP transport** (no live model).
- Zero new runtime dependencies (Node ≥20 global `fetch`).
</objective>

  <name>Task 2: Implement the tier-2 OpenAI-compatible adapter and wire it in</name>
  <files>src/watch/tier2.ts (option-a) or src/watch/tier-runner.ts (option-b), src/watch/tier-runner.ts, src/watch/index.ts</files>
  <action>
    Implement the adapter as a pure core + a thin effectful factory. Place it
    per the Task 1 decision (new tier2.ts for option-a; inline in tier-runner.ts
    for option-b). Public shapes:

    1. `export interface Tier2Config { baseURL: string; model: string; apiKey?: string }`

    2. PURE request builder:
       `export function buildTier2Request(set: WatchedFrameSet, question: string, config: Tier2Config): { url: string; init: RequestInit }`
       - url = `${config.baseURL.replace(/\/+$/, "")}/chat/completions`
         (document that baseURL must include the OpenAI `/v1` segment, e.g.
         `http://localhost:8080/v1`).
       - body (JSON.stringify):
         `{ model: config.model, temperature: 0, messages: [{ role: "user",
            content: toOpenAIContent(set, { header }) }] }`
         where `header` states the question and asks the model to answer from the
         frames/transcript in timeline order. Import `toOpenAIContent` from
         `../contract/index.js` — do NOT re-serialize by hand (it is the proven
         tier-2 wire shape, DESIGN §5 #2).
       - init.headers: `{ "Content-Type": "application/json" }`, plus
         `Authorization: "Bearer " + config.apiKey` ONLY when apiKey is set.
       - init.method = "POST"; init.body = the JSON string.

    3. PURE response parser:
       `export function parseTier2Answer(json: unknown): string | null`
       - return the text of `choices[0].message.content` when it is a non-empty
         string; if content is an array of parts, join the text parts; otherwise
         return null. Defensive: any missing/wrong-typed field → null (never throw).

    4. PURE config resolver (env bridge — the single env effect, isolated so
       Phase 7 can swap it for a real config surface):
       `export function resolveTier2ConfigFromEnv(env = process.env): Tier2Config | null`
       - reads `WATCH_TIER2_BASE_URL`, `WATCH_TIER2_MODEL`, `WATCH_TIER2_API_KEY`.
       - returns a Tier2Config only when BOTH baseURL and model are present and
         non-empty; otherwise null. apiKey optional. (This is the only env read.)

    5. EFFECTFUL factory:
       `export function createTier2Runner(deps?: { config?: Tier2Config | null; fetchImpl?: typeof fetch }): TierRunner`
       The returned async runner, per call:
       - resolve config: `deps.config !== undefined ? deps.config : resolveTier2ConfigFromEnv()`.
         If config is null → `return null` (escalate; unconfigured) WITHOUT calling fetch.
       - build the request via `buildTier2Request`.
       - `const f = deps?.fetchImpl ?? fetch;` then
         `try { const res = await f(url, init); if (!res.ok) return null;
            const json = await res.json(); const answer = parseTier2Answer(json);
            return answer === null ? null : { tier: 2, content: [{ type: "text", text: answer }],
              details: { tier: 2, model: config.model } };
          } catch { return null; }`
       - NEVER log config.apiKey or the Authorization header.

    6. Wire-in: set `defaultRunners[2]` to `createTier2Runner()` (the env-resolving,
       global-fetch default). Replace the old `tier2Runner` stub; keep a named
       export `tier2Runner = createTier2Runner()` so the barrel/back-compat export
       stays valid. Under option-a, tier-runner.ts imports `createTier2Runner`
       from `./tier2.js`.

    7. Docstrings: update the tier-runner.ts module docstring (and index.ts) so
       they no longer say tier 2 is "a stub/seam" or claim blanket purity —
       state that tier 2 is the network adapter (pure builder/parser + isolated
       fetch/env effect) while tier 1/3 + walk remain pure. Export the new public
       surface (`Tier2Config`, `buildTier2Request`, `parseTier2Answer`,
       `resolveTier2ConfigFromEnv`, `createTier2Runner`) from `src/watch/index.ts`.

    Avoid: adding any npm dependency (use global fetch); sending a video file or
    a `video_url` (frames-as-images is the proven path); per-model branching
    (one baseURL+model path); throwing out of the runner (always null on failure);
    changing extension.ts / walkTierChain (the async seam is already in place).
  </action>
