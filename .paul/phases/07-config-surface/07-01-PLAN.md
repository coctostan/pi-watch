<objective>
## Goal
Replace the tier-2 env bridge (`resolveTier2ConfigFromEnv` + raw `WATCH_TIER2_*` reads) with a single typed **config surface** for the `watch` tool, and fold a `fetch` timeout (`AbortSignal`) into the tier-2 adapter. The config resolves the tier-2 endpoint (`baseURL` + `model id` + optional `apiKey`), a default frame budget, a default resolution, and a fetch timeout — from defaults layered with environment values, with explicit per-call tool-param overrides. No committed secrets; zero new dependencies.

## Purpose
DESIGN §4 ("Adapter = `baseURL` + `model id`") and the PROJECT/AGENTS source-of-truth say *we own the sampling; model backends are thin OpenAI-compatible adapters configured by `baseURL` + `model id`, never code forks; cloud is optional, never required*. Phase 6 deliberately isolated the model's first env read in one function so Phase 7 could swap it for a real config surface without touching the adapter. This phase delivers that typed surface and closes the PETE carry (tier-2 `fetch` has no timeout — a slow/hung endpoint currently blocks until the runtime default).

## Output
- New pure config module `src/config/` (`config.ts` + `index.ts` barrel): `WatchConfig` type + a pure `resolveWatchConfig(env, overrides?)` resolver.
- `src/watch/tier2.ts`: `createTier2Runner` gains an optional `timeoutMs` that wires an `AbortSignal` timeout into the tier-2 `fetch`; behaviour on timeout = `null` (escalate to tier 3), never a throw through the host.
- `src/watch/extension.ts`: resolves the typed config once at registration, builds a config-driven tier-2 runner, and applies config defaults under per-call `WATCH_PARAMS` overrides.
- `src/watch/index.ts`: barrel updated (config surface re-exported; deprecated env bridge note).
- Tests: `test/config/config.test.ts` (new, resolver precedence + tier-2 mapping); `test/watch/tier2.test.ts` extended (timeout → null escalation).
- `.paul/phases/07-config-surface/07-01-SUMMARY.md` at completion.
</objective>

## AC-1: Typed config resolves from defaults + env with documented precedence
```gherkin
Given the environment sets WATCH_TIER2_BASE_URL, WATCH_TIER2_MODEL, and optionally WATCH_TIER2_API_KEY / WATCH_BUDGET / WATCH_RESOLUTION / WATCH_TIER2_TIMEOUT_MS
When resolveWatchConfig(env) runs
Then it returns a fully-resolved WatchConfig whose tier2 is a { baseURL, model, apiKey? } object, with budget, resolution, and fetchTimeoutMs taken from env when present and from built-in defaults otherwise
And when neither WATCH_TIER2_BASE_URL nor WATCH_TIER2_MODEL is set, config.tier2 is null (tier 2 "unconfigured" → the chain escalates to tier 3)
And explicit overrides passed to resolveWatchConfig(env, overrides) win over env values, which win over defaults
```

## AC-2: Extension wires the typed config; tool-call params override config defaults
```gherkin
Given the watch extension is registered with a resolved config that sets a tier-2 endpoint, a default budget, and a default resolution
When the watch tool executes for a question
Then sample() receives budget/resolution = the per-call WATCH_PARAMS value when provided, else the config default
And walkTierChain runs with a tier-2 runner built from config.tier2 (config-driven), not from a raw process.env read inside the adapter
And no behaviour of the frozen router (route.ts) or the pure walk core (walkTierChain) is changed
```

## AC-3: Tier-2 fetch honours a timeout and escalates on abort; gates stay clean
```gherkin
Given createTier2Runner is built with a configured endpoint and a timeoutMs
When the tier-2 fetch exceeds timeoutMs (or the AbortSignal fires)
Then the runner resolves to null (so walkTierChain escalates to tier 3) and never throws through the host
And the API key is never logged
And the change adds zero new dependencies, typecheck + build pass, and npm audit reports 0 vulnerabilities
```
