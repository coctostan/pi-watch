<objective>
## Goal
Ship the `/watch` slash command as the UX wrapper over the existing `watch` tool
primitive (DESIGN §1, §7 build-order step 5). A user types
`/watch <video-ref> <question>` and the command turns that into a `watch`
tool invocation, preserving all three tiers — crucially tier 3, whose sampled
frames must reach the **orchestrator model** as tool-result `ImageContent`
(DESIGN §5 Verified Fact #1).

## Purpose
Phase 8 closes the "tool primitive + command UX" pairing the project was built
around (PROJECT.md key decision: "build the `watch` tool as the primitive first;
the `/watch` command and batching wrap it"). The sampler, router, all three
tiers, and the typed config surface are already in place — this phase adds the
thin, ergonomic user-facing entry point on top, with zero changes to the frozen
pure core.

## Output
- `src/watch/command.ts` (new) — pure command core: argument parsing + invocation
  message builder + effect-injected `runWatchCommand`.
- `src/watch/extension.ts` (modified) — registers the `/watch` command alongside
  the existing `watch` tool, synchronously, in the same factory.
- `src/watch/index.ts` (modified) — export the new command surface from the barrel.
- `test/watch/command.test.ts` (new) — deterministic specs for the pure core +
  the effect-injected handler (stubbed notify/send; no pi runtime).
</objective>

## Key API facts (verified against installed types @earendil-works/pi-coding-agent@0.79.8)
- Command registration:
  `pi.registerCommand(name, { description?, getArgumentCompletions?, handler })`
  where `handler: (args: string, ctx: ExtensionCommandContext) => Promise<void>`.
  `args` is the RAW string after `/watch ` (the command must parse it itself).
- User-facing notifications: `ctx.ui.notify(message, "info" | "warning" | "error")`.
- Re-entering the agent loop: `pi.sendUserMessage(content: string, options?)` —
  ALWAYS triggers a turn; this is how the command hands work to the agent so the
  agent calls the `watch` tool and tier-3 frames land in the orchestrator's context.
- `registerCommand` is on `ExtensionAPI` (same object as `registerTool`); register
  synchronously in the existing factory (no await before registration — activation recipe).

<task type="checkpoint:decision" gate="blocking">
  <decision>How should the /watch command execute the watch pipeline?</decision>
  <context>
    The watch tool's universal fallback is tier 3: it hands sampled frames to the
    ORCHESTRATOR model as tool-result ImageContent (DESIGN §2, §5 Verified Fact #1).
    A slash-command handler returns void and can only `ctx.ui.notify(...)` — it has
    no channel to inject images into the model's context. So the execution strategy
    materially decides whether the command preserves all three tiers. This mirrors
    the deliberate forks at Phase 5 (ExtensionAPI type source) and Phase 7 (config
    module shape).
  </context>
  <options>
    <option id="option-a">
      <name>Delegate to the agent via pi.sendUserMessage (RECOMMENDED)</name>
      <pros>
        - Preserves ALL three tiers, especially tier 3: the agent calls the `watch`
          tool in the normal tool-call flow, so frame ImageContent reaches the
          orchestrator exactly as designed (DESIGN §5 #1).
        - Zero duplication: reuses the existing tool `execute` path untouched.
        - Thinnest possible command; keeps extension.ts a wiring boundary.
        - The agent can also pick budget/resolution from the question intent (router).
      </pros>
      <cons>
        - Indirect (the LLM performs the tool call rather than the handler).
        - Relies on the prompt steering the agent to the `watch` tool (mitigated by
          the existing promptSnippet/promptGuidelines on the tool).
      </cons>
    </option>
    <option id="option-b">
      <name>Direct pipeline invocation in the handler</name>
      <pros>
        - Deterministic: the handler runs sample → route → walkTierChain itself.
        - No dependence on the agent choosing the tool.
      </pros>
      <cons>
        - BREAKS tier 3: frames returned by walkTierChain are ImageContent meant for
          the orchestrator; a command handler can only notify text, so the universal
          visual fallback cannot reach the model. Defeats the core architecture.
        - Duplicates/forks the tool execute logic (config resolution, runner table),
          violating the single-effect-boundary convention.
      </cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b</resume-signal>
</task>

<task type="auto">
  <name>Task 2: Pure command core — parse + prompt builder + effect-injected runner (with tests)</name>
  <files>src/watch/command.ts, test/watch/command.test.ts</files>
  <action>
    Create `src/watch/command.ts` as a PURE module (no pi import, no I/O), mirroring
    the tier-runner.ts pure-core convention. Export:

    1. `WATCH_COMMAND_USAGE` — a constant usage string, e.g.
       `Usage: /watch <video-path-or-url> <question>`.

    2. `parseWatchCommand(args: string): WatchCommandParse` where
       `type WatchCommandParse =
          | { ok: true; ref: string; question: string }
          | { ok: false; usage: string }`.
       - Trim `args`. If empty → `{ ok: false, usage: WATCH_COMMAND_USAGE }`.
       - Split off the FIRST whitespace-delimited token as `ref`; the remaining
         trimmed text is `question`.
       - If `question` is empty (ref given but no question) →
         `{ ok: false, usage: WATCH_COMMAND_USAGE }`.
       - Otherwise `{ ok: true, ref, question }`. Total/never-throws.

    3. `buildWatchPrompt(ref: string, question: string): string` — a pure builder
       that produces the user message steering the agent to call the `watch` tool,
       e.g. `Use the watch tool to answer this question about the video "${ref}": ${question}`.
       (Keep it one clear sentence; do not embed tool params — the router/config own
       budget/resolution defaults.)

    4. `runWatchCommand(args, effects)` where
       `interface WatchCommandEffects {
          notify: (message: string, level: "info" | "warning" | "error") => void;
          send: (content: string) => void;
        }`.
       - Parse via `parseWatchCommand`.
       - On `{ ok: false }`: call `effects.notify(usage, "warning")` and return; do NOT send.
       - On `{ ok: true }`: call `effects.send(buildWatchPrompt(ref, question))`; do NOT notify an error.
       This is the injected-effects pattern (mirrors tier2's injected `fetchImpl`),
       so the whole command logic is unit-testable without the pi runtime.

    Then create `test/watch/command.test.ts` (Vitest, import from `../../src/watch/index.js`):
       - parseWatchCommand: ref+question happy path; whitespace trimming; empty input;
         ref-only (missing question); URL ref; path-with-no-question. Assert the
         usage string on the not-ok cases.
       - buildWatchPrompt: contains the ref and the question verbatim.
       - runWatchCommand: with stub `notify`/`send` (capture calls) — valid input
         sends exactly once and never error-notifies; invalid input notifies once at
         "warning" with the usage string and never sends.

    Do NOT touch the frozen core (sampler/router/config/tier-runner/tier2). Add
    docstrings explaining the pure-core/effect-boundary split.
  </action>
  <verify>npm test -- command  &&  npm run typecheck</verify>
  <done>AC-1 + AC-2 satisfied: parser and runner behave per spec; new specs green.</done>
</task>
