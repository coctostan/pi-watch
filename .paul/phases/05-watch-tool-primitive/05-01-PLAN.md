<objective>

## Goal
Wire the load-bearing `watch` primitive end-to-end: a `watch` **pi custom tool** that takes a
video `ref` + `question`, runs `sample()` → `routeContextFromSet()` → `route()`, walks the
ordered tier chain, and returns an answer. Tiers 1–2 (real model adapters) are seams that
report "unavailable" and escalate; **tier 3 (frames-into-context) is fully implemented** —
it hands the sampled frames back to the orchestrator model as tool-result `ImageContent`,
which is the DESIGN §2 universal fallback and needs no external model call.

## Purpose
This is the project's load-bearing seam (PROJECT.md key decision: "Build the `watch` tool as
the primitive first; command + batch wrap it"). It is the first consumer that composes the
three stable surfaces shipped in Phases 2–4 (`sample`, `route`, contract). After this plan,
the agent can actually watch a video and answer a question via tier 3, with tiers 1–2 ready
to drop in during Phase 6 behind a stable runner seam.

## Output
- `src/watch/tier-runner.ts` — pure tier-walk core + tier-3 frame→tool-result serializer + tier 1/2 stubs.
- `src/watch/extension.ts` — the pi extension factory that registers the `watch` tool (effect boundary).
- `src/watch/index.ts` — barrel export for the watch module.
- `test/watch/tier-runner.test.ts` — unit specs for the pure walk + tier-3 serializer.
- `package.json` — `pi.extensions` manifest entry + (pending decision) the `ExtensionAPI` type source.
- `.paul/phases/05-watch-tool-primitive/05-01-SUMMARY.md` — at UNIFY.
</objective>

## AC-1: Tier walk escalates through stubs and always resolves at tier 3
```gherkin
Given a RoutingDecision whose tiers chain is [2, 3] (or [1, 2, 3])
  And tier-1 and tier-2 runners that report "unavailable" (return null)
When walkTierChain runs over the chain
Then it skips the unavailable tiers in order
  And returns the tier-3 result
  And the returned result reports tier === 3
```

## AC-2: Tier walk stops at the first available tier
```gherkin
Given a RoutingDecision whose tiers chain is [2, 3]
  And a tier-2 runner that returns a result
When walkTierChain runs over the chain
Then it returns the tier-2 result without invoking tier 3
  And the result reports tier === 2
```

## AC-3: Tier-3 serializer hands frames to the orchestrator as ImageContent
```gherkin
Given a WatchedFrameSet with N ordered frames (mediaType image/png) and a question
When the tier-3 runner builds the tool-result content
Then the content contains one image part per frame in timeline order
  And each image part is { type: "image", data: <frame.imageBase64>, mimeType: <frame.mediaType> }
  And a leading text part states the question, frame count, and that tier 3 (frames-into-context) is in use
  And each frame's mm:ss timestamp appears as text adjacent to its image
  And when transcript segments exist they are included as text (and omitted when transcriptSource is "none")
```

## AC-4: The watch tool is registered and invocable, and the pipeline runs end-to-end
```gherkin
Given a pi session that loads the built watch extension
When the extension factory runs
Then pi.getAllTools() includes a tool named "watch" with a non-empty promptSnippet and TypeBox parameters {ref, question, ...}
  And invoking execute({ ref, question }) calls sample() → routeContextFromSet() → route() → walkTierChain()
  And returns { content } whose parts are valid TextContent | ImageContent
  And for a purely visual question with no transcript (route [2,3]) the answer resolves at tier 3 (frames returned)
```

  And dist/watch/extension.js exists and default-exports a function
