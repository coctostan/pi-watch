<objective>
## Goal
Add batching as the final v0.1 capability (DESIGN §7 step 6): watch many videos in
one call by fanning the existing sample → route → walkTierChain pipeline out over a
list of `{ ref, question }` items in parallel, then aggregating the results. Ship it
as a new `watch_batch` tool over a pure, unit-testable `runWatchBatch` core. Tiers
1/2 (text answers) aggregate into one bounded text tool-result; tier-3 (frames) items
are surfaced for individual `/watch` rather than inlining many videos' frames —
honoring DESIGN §5/§9 ("batching is trivial for tiers 1/2 → Promise.all; only tier-3
batch needs subagent fan-out — defer it").

## Purpose
Closes the v0.1 build order (PROJECT.md key decision: "build the watch tool as the
primitive first; the command and batching wrap it"). The sampler, router, all three
tiers, config surface, the `watch` tool, and the `/watch` command are all in place to
batch over — batching is the last wrapper. Completing it triggers milestone
completion (Route B), not another phase.

## Output
- `src/watch/batch.ts` — pure batch core (`runWatchBatch`, `BatchItem`,
  `BatchItemResult`, `BatchResult`, `WatchItemProcessor`).
- `test/watch/batch.test.ts` — deterministic specs (stubbed per-item processor; no
  ffmpeg, no network).
- `src/watch/extension.ts` — registers `watch_batch` synchronously alongside the
  existing `watch` tool, wiring the real sample → route → walkTierChain pipeline as
  the injected per-item processor (additive only).
- `src/watch/index.ts` — exports the new batch surface from the barrel.
</objective>

<acceptance_criteria>

## AC-1: Parallel fan-out with per-item error isolation
```gherkin
Given a list of N watch items and an injected per-item processor
When runWatchBatch is invoked
Then every item is processed concurrently (Promise.allSettled, not sequential)
And each item yields a BatchItemResult preserving its input index/ref/question
And a single item that rejects becomes a status:"error" result with the message
And does NOT abort the other items (no all-or-nothing failure)
```

## AC-2: Text-tier aggregation; tier-3 batch deferred (bounded, text-only result)
```gherkin
Given processed items where some routed to tier 1/2 (text) and some to tier 3 (frames)
When the batch core aggregates the results into tool-result content
Then tier-1/2 items contribute their text answer parts under a per-item section
And tier-3 items contribute a short deferral note ("routed to the frames tier;
  run /watch <ref> individually") instead of inlining their frame image parts
And the aggregated content contains NO image parts (bounded text-only output),
  realizing DESIGN §5/§9 "Promise.all over tiers 1/2; defer tier-3 fan-out"
```

<task type="auto">
  <name>Task 2: Pure batch core (runWatchBatch) + deterministic tests</name>
  <files>src/watch/batch.ts, test/watch/batch.test.ts</files>
  <action>
    Create `src/watch/batch.ts` — a PURE, pi-free fan-out + aggregation core (mirrors
    tier-runner.ts: imports contract/router/tier-runner shapes as TYPES only; no
    sample()/route()/network here). Define and export:
    - `interface BatchItem { ref: string; question: string }`
    - `type WatchItemProcessor = (item: BatchItem) => Promise<TierResult>` — the
      injected per-item pipeline (the extension boundary supplies the real
      sample → route → walkTierChain in Task 3).
    - `interface BatchItemResult { index: number; ref: string; question: string;
      status: "ok" | "error"; tier?: Tier; content?: WatchContentPart[];
      details?: Record<string, unknown>; error?: string }`
    - `interface BatchResult { items: BatchItemResult[]; content: WatchContentPart[] }`
    - `async function runWatchBatch(items, deps: { processItem }): Promise<BatchResult>`

    Behavior:
    - Fan out with `Promise.allSettled(items.map(processItem))` — concurrent, and one
      rejected item must NOT abort the batch (AC-1). (Deliberate refinement of DESIGN's
      "Promise.all": allSettled gives per-item error isolation while keeping the same
      parallel fan-out intent.)
    - Map each settled outcome to a BatchItemResult preserving the input index/ref/
      question. Fulfilled → status:"ok" with tier/content/details from the TierResult.
      Rejected → status:"error" with the Error message (use `err instanceof Error ?
      err.message : String(err)`).
    - Aggregate `content` (pure, AC-2): a leading text part ("Watched N video(s);
      results below."), then per item a text section header
      (`── [index] ref — question`) followed by:
        * tier 1 or 2 → that item's text parts (the answer).
        * tier 3 → a SINGLE deferral text part: routed to the frames tier (tier 3);
          run `/watch "<ref>"` individually to bring its frames into context (batch
          frame fan-out deferred — DESIGN §5/§9). Do NOT push the frame image parts.
        * error → a text part with the error message.
      The aggregated `content` MUST contain only `{ type: "text" }` parts (no images).
    - Empty `items` → a single text part stating nothing was provided; `items: []`.
    - Add a file docstring citing DESIGN §5/§7 step 6/§9 and the pure-core/
      effect-boundary convention.

    Create `test/watch/batch.test.ts` (vitest, mirror tier-runner.test.ts style):
    - A `makeTierResult(tier, text?)` helper and stubbed `processItem`s; NO real
      sample()/route()/walkTierChain, NO ffmpeg, NO network.
    - Specs: (1) parallel fan-out preserves order/index; (2) one rejecting item →
      status:"error", others still ok (isolation); (3) tier-1/2 text aggregated into
      the combined content; (4) tier-3 item emits the deferral note and NO image part
      (assert no part has type "image"); (5) mixed batch (1 + 2 + 3 + error) aggregates
      correctly; (6) empty items → graceful single-text result.
  </action>
  <verify>npm run typecheck && npx vitest run test/watch/batch.test.ts</verify>
  <done>AC-1 + AC-2 satisfied: parallel, error-isolated fan-out with bounded
  text-only aggregation (tier-3 deferred); new specs green.</done>
</task>
