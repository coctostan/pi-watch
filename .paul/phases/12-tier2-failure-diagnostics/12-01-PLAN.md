<objective>
## Goal
Make tier-2 failures legible. Replace the single silent `null` that `createTier2Runner` returns for every failure mode with a structured tier-2 diagnostic — `unconfigured`, `http-error` (with status), `empty-answer`, `timeout`, or `network-error` — that is surfaced in the `watch` (and `watch_batch`) tool-result `details`, WITHOUT changing the null→tier-3 escalation behavior, the OpenAI-compatible wire shape, or the model-agnostic adapter contract.

## Purpose
v0.2 is "Tier 2, For Real": `/watch` should answer from the real model and **degrade legibly**. Today `src/watch/tier2.ts` collapses five distinct failures (no config / non-2xx / empty-or-garbled answer / timeout / network error) into one bare `return null`, and `extension.ts` builds `details` with no record of why tier 2 was skipped. A developer or user cannot tell "tier 2 unconfigured" from "endpoint down" from "model returned garbage" — they only see a tier-3 frame answer. This phase pays down that pre-existing silent-failure debt (carried since v0.1, deferred from Phase 11) so failures are diagnosable while the universal tier-3 fallback still works exactly as before.

## Output
- A `Tier2Diagnostic` type and per-failure-reason emission in `src/watch/tier2.ts`.
- The chosen diagnostic-plumbing mechanism wired from the tier-2 runner to the tool-result `details` at the extension boundary (`src/watch/extension.ts`), via `src/watch/tier-runner.ts` only if the decision requires it.
- A new public export for `Tier2Diagnostic` in `src/watch/index.ts`.
- Deterministic unit coverage in `test/watch/tier2.test.ts` (and `test/watch/tier-runner.test.ts` if the core contract changes) proving each reason is reported AND that escalate-to-tier-3 is unchanged.
- `docs/TIER2-SETUP.md` updated to document the `details.tier2` failure surface and how to read it when a watch silently used tier 3.
- `.paul/phases/12-tier2-failure-diagnostics/12-01-SUMMARY.md` at UNIFY.
</objective>

## AC-1: Each tier-2 failure mode produces a distinct, safe diagnostic
```gherkin
Given the tier-2 runner is invoked through its production builder/parser path with diagnostics enabled
When tier 2 fails because it is unconfigured, returns a non-2xx status, returns an empty/garbled answer, times out, or hits a network error
Then a structured tier-2 diagnostic is produced naming the corresponding reason (unconfigured | http-error | empty-answer | timeout | network-error), an http-error diagnostic includes the numeric status, and no diagnostic message contains the api key, Authorization header, or full request body
```

## AC-2: Escalation to tier 3 is byte-for-byte preserved
```gherkin
Given the existing tier-2 escalation contract (a failing tier-2 runner returns null and walkTierChain advances to the next tier, ending in the total tier-3 runner)
When any tier-2 failure occurs with diagnostics enabled
Then the runner still escalates exactly as before (the walk still resolves at tier 3 with the same content), no exception is thrown through the host, and the unconfigured path still performs no network call
```

## AC-3: The diagnostic reaches the tool-result details
```gherkin
Given a watch tool call whose router chain includes tier 2 but tier 2 does not answer (so tier 3 answers)
When the watch (or watch_batch) tool returns
Then its details record the tier-2 diagnostic (reason and, when applicable, http status / short message) so an operator can tell why tier 2 was skipped, while a successful tier-2 answer (final tier === 2) records no failure diagnostic
```

## AC-4: The success path and wire shape are unchanged
```gherkin
Given a well-formed tier-2 response from the OpenAI-compatible endpoint
When the runner succeeds
Then it returns the same tier-2 TierResult as before (no diagnostic), buildTier2Request/parseTier2Answer remain pure and model-agnostic with no per-model branch, and the existing deterministic and opt-in live tests still pass
```

<task type="checkpoint:decision" gate="blocking">
  <decision>How does the tier-2 failure diagnostic flow from the runner out to the tool-result `details`, given the runner currently returns a bare `null` and `walkTierChain` treats `null` as "escalate"?</decision>
  <context>
    The diagnostic must be attributable per tool call (and per `watch_batch` item — the SAME runner instance handles N parallel items today). It must not change the `null === escalate` invariant in `walkTierChain`. This is a contract-level choice (AGENTS.md: "Ask First before changing architecture patterns"), so it is decided before implementation. Both options preserve escalation and keep `buildTier2Request`/`parseTier2Answer` pure; they differ in blast radius and how "typed" the diagnostic is.
  </context>
  <options>
    <option id="option-a">
      <name>Boundary collector — runner gains an optional onDiagnostic callback; the extension builds a fresh collector + tier-2 runner per call</name>
      <pros>
        - Lowest blast radius: `TierRunner` and `walkTierChain` stay byte-for-byte unchanged (and so do `test/watch/tier-runner.test.ts` escalation tests) — `impact` flagged the extension boundary as the only depth-1 dependent.
        - Per-call / per-batch-item attribution is clean: each `execute` (and each batch item processor) constructs a fresh collector + `createTier2Runner({ config, timeoutMs, onDiagnostic })`, runs `walkTierChain`, then reads the collector and merges it into `details`.
        - The side effect (mutable collector) stays at the effect boundary; the pure builders/parser stay pure. `null` keeps its exact meaning.
      </pros>
      <cons>
        - The diagnostic is a side-channel callback rather than a first-class value of the tier-walk result.
        - The extension constructs a tier-2 runner per call instead of once (cheap, but a small change to the current "build runners once" pattern).
      </cons>
    </option>
    <option id="option-b">
      <name>Typed core — widen the tier-walk so runners report a tagged skip and walkTierChain returns accumulated diagnostics</name>
      <pros>
        - The diagnostic is a first-class, typed part of the tier-walk result (e.g. `walkTierChain` returns `{ ...TierResult, diagnostics }`); no mutable side channel.
        - Diagnostics are uniformly available for any future diagnostic-bearing tier, not just tier 2.
      </pros>
      <cons>
        - Larger blast radius: changes the `TierRunner` args/return contract and `walkTierChain`, and forces updates to the tier-1/tier-3 runners and to `test/watch/tier-runner.test.ts` (the `toBeNull()` / `result.tier === 3` anchors).
        - More surface to get wrong while the load-bearing invariant ("null === escalate, chain ends in total tier 3") must stay exactly intact.
      </cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b</resume-signal>
</task>
