<objective>
## Goal
Build the **router** as its own pure, deterministically-tested decision unit (DESIGN §2, PRD "Router (tier selection)"): given a *question* plus the availability context derived from a `WatchedFrameSet`, decide (a) which tier answers, (b) the ordered escalation chain ending in the universal tier-3 fallback, and (c) the frame resolution (high only for on-screen-text / OCR questions). No model calls, no I/O — routing is a design artifact, not an afterthought of the tool.

## Purpose
"Watching" a video = **pick the cheapest path that works** (PROJECT core value; DESIGN §2). Tier discipline is a project boundary (AGENTS.md): pick the cheapest tier that answers the question; tier 3 is the universal fallback. The router is the load-bearing decision that enforces this, and the PRD flags router quality as the actual un-de-risked hard part — so it gets its own deterministic test surface ("assert routes, not just answers"), independent of any probabilistic model output. Phase 5 (`watch` tool) and Phase 6 (tier adapters) consume this decision; they never re-derive routing.

## Output
- `src/router/route.ts` — pure router: `Tier`/`QuestionIntent`/`RouteContext`/`RoutingDecision` types, `classifyQuestion()`, `route()`, and `routeContextFromSet()` (the WatchedFrameSet → context adapter).
- `src/router/index.ts` — the router subsystem's public surface.
- `test/router/route.test.ts` — deterministic route assertions (AC-1…AC-5).
</objective>

<acceptance_criteria>

## AC-1: "What's said" question with a transcript routes to tier 1
```gherkin
Given a question about spoken content (e.g. "What does the narrator say about X?")
  And a RouteContext where hasTranscript is true
When route() is called
Then the decision's primaryTier is 1 (transcript)
  And tiers is the ordered escalation chain [1, 2, 3]
  And resolution is "low"
```

## AC-2: Spoken question with NO transcript escalates past tier 1
```gherkin
Given a question about spoken content
  And a RouteContext where hasTranscript is false
When route() is called
Then tiers does NOT start with tier 1 (tier 1 cannot answer without a transcript)
  And tiers is [2, 3] (conservative escalate-on-insufficiency)
  And routeContextFromSet(set) yields hasTranscript=false when set.transcript is empty / transcriptSource is "none", and true when transcript segments exist
```

## AC-3: Temporal/visual question routes to the video tiers with tier 3 as terminal fallback
```gherkin
Given a question needing temporal or visual reasoning (e.g. "What color comes after red?")
  And any RouteContext
When route() is called
Then tiers is [2, 3] (native video, then frames-into-context)
  And the last element of tiers is always 3 (universal fallback)
  And resolution is "low"
```

## AC-4: On-screen-text question triggers the high-res / OCR path
```gherkin
Given a question about text shown on screen (e.g. "What does the sign say?" / "Read the title card")
When route() (or classifyQuestion()) is called
Then intent is "on-screen-text"
  And resolution is "high"
  And tiers is [2, 3] (a vision tier reads the high-res frames)
```

## AC-5: Routing is pure and deterministic
```gherkin
Given identical inputs
When route() is called multiple times
Then it returns an identical decision every time (no randomness, no clock, no I/O)
  And every non-empty tiers chain ends in tier 3
  And route() does not mutate its inputs
  And classifyQuestion() depends only on the question string
```

    Types:
    - `export type Tier = 1 | 2 | 3;` — DESIGN §2 tiers (1 transcript / 2 native video / 3 frames-into-context).
    - `export type QuestionIntent = "spoken" | "visual" | "on-screen-text";`
    - `export interface RouteContext { hasTranscript: boolean; }` — the post-sample availability the tier choice depends on. Keep it minimal and derivable from a WatchedFrameSet.
    - `export interface RoutingDecision { intent: QuestionIntent; resolution: ResolutionTier; tiers: Tier[]; primaryTier: Tier; rationale: string; }`
      Import `ResolutionTier` as a type from `../contract/index.js`.

    `export function classifyQuestion(question: string): { intent: QuestionIntent; resolution: ResolutionTier }`:
    - Pure, question-only. Lowercase + scan for keyword groups (define small `readonly string[]` constants):
      - on-screen-text markers FIRST (highest specificity): e.g. "on screen", "on-screen", "text", "caption" (when about shown text), "written", "read the", "what does it say", "what does the sign", "sign say", "label", "subtitle", "title card", "logo", "watermark", "ocr", "displayed". → intent "on-screen-text", resolution "high".
      - spoken markers: "say", "said", "says", "speak", "spoken", "mention", "talk", "discuss", "dialogue", "narrat", "audio", "hear", "quote", "word for word", "transcript", "according to". → intent "spoken", resolution "low".
      - else → intent "visual", resolution "low".
    - Order matters: check on-screen-text before spoken before defaulting to visual (a question can contain "say" inside "what does the sign say" — on-screen-text must win). Document the precedence in a comment and cover it in tests.

    `export function route(args: { question: string; context: RouteContext }): RoutingDecision`:
    - Call `classifyQuestion(args.question)` for intent + resolution.
    - Tier selection (deterministic):
      - intent "spoken" + context.hasTranscript === true  → tiers = [1, 2, 3]
      - intent "spoken" + context.hasTranscript === false → tiers = [2, 3]   (escalate past tier 1)
      - intent "visual"                                    → tiers = [2, 3]
      - intent "on-screen-text"                            → tiers = [2, 3]
    - `primaryTier = tiers[0]`. Build a short human-readable `rationale` string from intent + transcript availability.
    - INVARIANT to uphold in code: every returned `tiers` is non-empty and its last element is `3` (universal fallback). Do not mutate `args`.

    `export function routeContextFromSet(set: WatchedFrameSet): RouteContext`:
    - Import `WatchedFrameSet` as a type from `../contract/index.js`.
    - `hasTranscript = set.transcript.length > 0 && set.source.transcriptSource !== "none"`.
    - Pure; reads only, returns `{ hasTranscript }`. This is the seam that makes the router "route over the WatchedFrameSet sample() produces" while keeping `route()` testable with hand-built contexts.

    Add a top module JSDoc explaining: pure decision layer (DESIGN §2), tier discipline (cheapest tier that works; tier 3 universal fallback), and that escalation is expressed as the ordered chain the watch tool (Phase 5) walks — confidence-based escalation from model answers is explicitly out of scope here.

    Create `src/router/index.ts` — re-export everything public:
    `export { route, classifyQuestion, routeContextFromSet, type Tier, type QuestionIntent, type RouteContext, type RoutingDecision } from "./route.js";`
    Use `.js` import specifiers (NodeNext/ESM, matching the rest of src). Do NOT touch package.json `main` (router stays internal until Phase 5 wires it).

## SCOPE LIMITS
- **Pure decision only.** No model/adapter calls, no HTTP, no ffmpeg, no transcript fetching, no `sample()` invocation in the router or its tests. The router returns a *decision*, not an answer.
- **No tier adapters / no execution of the chain** — walking the escalation chain and any confidence-based "the answer was insufficient → escalate" logic belongs to the `watch` tool (Phase 5) / tier adapters (Phase 6).
- **No config surface** — tier order/enablement, budget, and resolution *thresholds* as user config are Phase 7. v0.1 router uses the fixed policy above.
- **No new dependencies.** Pure TypeScript over existing contract types.
- Heuristic question classification is intentionally simple keyword/precedence matching for v0.1 (PRD: "conservative escalate-on-insufficiency policy"); richer classification is a later refinement, not this plan.
