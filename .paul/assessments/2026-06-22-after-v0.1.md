## Findings

### Vision alignment

Core Value = "cheapest-path-that-works video understanding for the agent — local-first, model-agnostic, never dependent on a single cloud provider." v0.1's nine phases map cleanly onto that vision: sampler → router → `watch` tool → three tier adapters → config → `/watch` command → `watch_batch`. No drift; every shipped phase serves the stated value.

**One material caveat against the vision:** the "cheapest path" promise is structurally present but not yet *real* at runtime. The router will emit `[1,2,3]` for spoken-content questions, but tier 1 (transcript) is a stub — `fetchTranscript` returns `source: "none"` for every input (local files: Whisper deferred; URLs: yt-dlp caption fetch deferred). In practice every question escalates straight to tier 2/3. The cheapest tier exists in the architecture but never fires. This is the single biggest gap between the design seed (DESIGN §2 tier 1 = "yt-dlp captions, else Whisper") and shipped behavior.

### Trajectory

Strong convergence, not sprawl. Linear 9-phase march, each phase a small PR (squash-merged #1–#11), all merged to main. Decision log (PROJECT.md Key Decisions) shows discipline: decisions are made once, marked Active/Validated/Superseded/Deferred, and not re-litigated (e.g. the env-bridge → config surface supersession is explicit; Phase 8 agent-delegation and Phase 9 `watch_batch`-as-wrapper are clean one-time calls). Deferred debt is *tracked, not leaking*: tier-3 fan-out, file-based config, `/watch` flags, TypeBox peer-dep cleanup all carried forward explicitly. Healthy.

### Codebase reality

~4,489 LoC across `src` + `test`. Largest files are tests (tier2 365, tier-runner 310, batch 213) — test-heavy, a good sign. Largest source files are reasonable and single-concern: `extension.ts` 279 (effect boundary), `tier-runner.ts` 250 (pure walk), `effects.ts` 241, `watched-frame-set.ts` 223. **Zero** TODO/FIXME/HACK markers. Clean pure-core/effect-boundary separation is consistently applied module by module. No obvious dead code or duplication. The codebase is small, legible, and honest.

### Buildable/runnable reality

- `npm run build` (tsc): clean, no errors.
- `npm test` (vitest): **125/125 passing**, 11 files, ~0.9s.
- `npm audit`: **0 vulnerabilities**.
- Gap: tests are pure-unit + one ffmpeg golden-clip integration test. There is **no live pi-runtime smoke test** for `watch`/`watch_batch` (does the registered tool actually activate + return ImageContent to a real orchestrator?), and **no dedicated CI workflow** — the only merge gate is Socket. So "it builds and unit-tests pass" is well-proven; "it works end-to-end in a live pi session against a real tier-2 endpoint" is asserted by Phase-1/spike evidence but not continuously verified.

### Data & outputs

{partial: no benchmark/eval outputs exist yet} The only "outputs" are the test suite and a golden-clip integration fixture (`sample.integration.test.ts`). There is **no eval harness** measuring the PRD's stated definition-of-done dimensions — golden-clip *answer* correctness, asserted routes, enforced frame budget, graceful degradation — as a repeatable, trend-able artifact. Routing and budget are unit-asserted; actual answer quality per tier is not measured anywhere. No regression baseline exists for "did a sampler/router change make answers worse."

## Strategic Options

- **A. Make tier 1 real (transcript ingestion)** — Implement yt-dlp caption fetch (URLs) and/or Whisper transcription (local files), merged onto the existing timeline seam (`mergeTranscript` already exists). *Cost:* medium (one effect + parsing; new external dep `yt-dlp`/whisper). *Risk:* low-medium (network/process boundary already patterned in `effects.ts`). *Value:* high — this is what makes "cheapest path" true; unlocks the whole tier-1 branch the router already routes to. *Deps:* none internal.
- **B. Eval / correctness harness** — A repeatable harness over golden clips that asserts answer correctness per tier, route selection, budget enforcement, and graceful degradation, emitting a trend-able results artifact. *Cost:* medium-high. *Risk:* low. *Value:* high for confidence and for safely iterating sampler/router; turns the PRD's DoD into a living gate. *Deps:* benefits from A (more tiers to actually exercise).
- **C. Productionization / release hardening** — Dedicated CI workflow (test+build+audit gate, not Socket-only), live pi-runtime smoke test, package/publish polish, docs/README for installers. *Cost:* low-medium. *Risk:* low. *Value:* medium-high — closes the "works in a real session / safe to ship & install" gap. *Deps:* none.
- **D. Tier-3 batch fan-out + richer surfaces** — Subagent fan-out for frames-for-many-videos, `/watch-batch` command, `/watch` budget/resolution flags, file-based config. *Cost:* medium-high (subagent orchestration is the hard, deferred part). *Risk:* medium (context-blowup is exactly why it was deferred). *Value:* medium — extends reach but not the core promise. *Deps:* stable single-video path (have it).

## Ranked Recommendation

**Recommended next target (non-authoritative):** **Option A — make tier 1 (transcript) real**, ideally paired with a thin slice of **Option C** (CI gate + live smoke) as hardening.

**Why this ranks first:** The most honest reading of v0.1 is "the cheapest-path architecture is complete, but the cheapest path doesn't run yet." Tier 1 is the literal embodiment of the project's Core Value ("cheapest path that works", "model-agnostic" — transcript needs no model at all). The router already routes to it; `mergeTranscript` already accepts it; the seam is built and waiting. Implementing it is the highest value-to-cost move and the one most aligned with the stated vision. Option B is the natural *second* milestone (you want real tiers before you build the harness that grades them), and a light dose of C de-risks shipping without pulling a full productionization milestone forward. D is genuinely deferrable — it extends reach, not the core promise.

> Non-authoritative: this recommendation informs the milestone discussion but does not decide it. Only a future approved PLAN authorizes implementation.
